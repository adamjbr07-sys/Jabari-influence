## Problem Statement
The creator (@adam.jbrr) wants to grow from ~1,650 TikTok followers and ~1,850 Instagram
followers to 10,000 on BOTH platforms by 2026-08-31. He grew before on viral videos and wants
to repeat it, bigger. The real bottleneck is not idea supply (he already has an idea + hook +
caption tool); it is conversion and cadence: "half ideas, nothing good enough to post," slow
ideation, slow editing. His own key observation: "the videos I put least effort into anyway
get more views." His honest framing: "I just need proper organization."

## What Makes This Cool
Content OS becomes a content production line that runs on his own results: idea bank ->
make-it-shootable -> daily film queue -> post -> log result -> clone winners, with TikTok and
Instagram as two separate scoreboards. It does not pretend to crack the algorithm; it tracks
which of HIS posts grow HIM on each platform and tells him to make more like them. The
"generate tomorrow's batch" button hands him the next 3 ranked videos on demand.

## Constraints
- Hard deadline: 2026-08-31 (~2 months).
- Two platforms, two scoreboards. Four lanes: Arab/Muslim, gym/fitness, engineering/student, vlogs.
- Builds on the existing Next.js 16 app: idea generator, hook/caption generator, trends route,
  Claude via subscription auth (Agent SDK). Reuse, do not rebuild.
- No social-platform scraping (ToS, brittle, account risk).
- Claude runs on the host's Claude subscription via the `claude` CLI subprocess (Node host,
  not Edge/serverless). Per decision D6, this runs on his laptop on demand (button), so
  subscription auth works as-is with no server-token gymnastics.

## Premises (agreed in office-hours)
1. Needs half-idea -> shootable -> posted, fast. The gap is conversion and cadence, not ideas.
2. Less editing, not more. Edit-time cap is a guardrail. NOTE (eng review): "least effort
   caused more views" is a correlation, not proven cause — the cap is sensible either way, but
   the winner score (D5) is what actually identifies what works, not edit time.
3. The machine is a production line, not a dashboard. The ONE allowed indicator is a follower
   trajectory "ahead/behind pace" line (see Success Criteria).
4. Per-platform winner loops, enforced at the data level (D3: per-platform `posts` records).
5. Tracking is manual/light: a daily follower log + per-post views logged at ~48h, refreshed
   at ~7d (consistent window so winner comparisons are fair). IG auto-ingest is conditional (B0).
6. 10k on both by August is the north star, not the build's pass/fail bar (see Success Criteria).

## Eng-Review Decisions (D1-D9, 2026-06-26)
- **D1 — DB in Phase A.** localStorage is per-browser; he uses phone + laptop. A shared store
  is the foundation of Phase A, not deferred. Winner-loop + briefing rank against one source of truth.
- **D2 — Supabase Postgres.** He knows it; gives a dashboard and sets up future charts.
  FOOTGUN: his shell exports `SUPABASE_URL`, so local dev must start with `env -u SUPABASE_URL`.
  The button briefing must fail gracefully when Supabase is unreachable.
- **D3 — Data model: ideas + posts (1:many) + follower_logs.** One idea -> many posts (one per
  platform per publish). `status`, `platform`, `views`, `postedAt`, `editMinutes`, `isWinner`
  live on `posts`, NOT `ideas`. Idea text/hook/caption/format/lane/draft-link live on `ideas`
  (DRY). **Follower history is its own table** `follower_logs(platform, date, count)` — the
  source of truth for growth and the pace line. Per-post follower attribution is DERIVED (a
  best-effort "candidate" from the daily delta on the post's publish day), stored as
  `posts.followerDeltaCandidate`, NOT authoritative. This resolves the earlier ambiguity
  (table vs denormalized): both exist, with `follower_logs` authoritative.
- **D4 — Add Vitest** for the pure ranking/threshold logic; keep Playwright for E2E.
- **D5 — Follower-aware winner score.** Log follower count daily per platform; attribute daily
  delta to posts (fuzzy: "which post spiked that day"). Rank winners by a blend that weights
  follows, not pure views. Optimizes the goal, not the proxy.
- **D6 — Button-driven batch, not scheduled.** "Generate tomorrow's batch" button on his laptop.
  Deletes the always-on-host + headless-subscription-auth risk. Scheduled push deferred (TODO).
- **D7 — Configurable per-platform daily target** (default 1-2/day/platform). One film can become
  two posts (cross-post). Drives the pace line.
- **D8 — Backfill 30-50 recent posts/platform** at launch (paste/import form: views, date, lane,
  format, hook type, follower delta if known). No cold start; ranking is live on day one.
- **D9 — Lightweight media field:** a "draft link / note" per idea/post (CapCut/Drive/Photos
  link). Full media/asset storage is OUT of scope (see NOT in scope).

## Data Flow (Phase A)
```
backfill (30-50 posts/platform) ─┐
                                 v
idea (Claude/generator) ── make-shootable (Claude) ── status: shootable
                                 │
                                 v
        queue (per-platform daily target, D7) ── film (raw, edit-capped)
                                 │
                                 v
   post -> one `posts` row per platform ── log views@48h/7d + daily follower delta
                                 │
                                 v
   winnerThreshold(posts, platform)  [follower-aware, D5]
       N<3: none | 3<=N<10: 2x median(N) | N>=10: 2x trailing-10 median
                                 │  winners
                                 v
   "make 3 more like this" (constrained: change angle/audience/opening/payoff; anti-fatigue)
       -> clones land as `shootable`, linked to source idea
                                 │
                                 v
   "generate tomorrow's batch" button -> rank by winners, cold-start -> lane rotation + trends
```

## Success Criteria
- North star (aspiration, NOT pass/fail): 10k followers TikTok AND Instagram by 2026-08-31.
  Depends on his on-camera performance and the algorithms; the tool cannot guarantee it.
- Growth math (so "cadence" isn't vibes): ~8k+ net follows/platform in ~2 months ≈ ~130/day/
  platform. The pace line shows required-vs-actual; the daily target (D7) is tuned to it.
- Definition of done (controllable, the real pass/fail bar):
  - Cadence: a sustained posts/day/platform streak hitting the D7 target (the ONE surfaced indicator).
  - Throughput: ideas reaching "posted" per week.
  - Winner rate + clone latency: posts above the follower-aware winner score per week, and how
    fast a winner gets 3 constrained clones shipped.
  - Edit-time cap honored: median logged edit-minutes <= 20 (dropped if not logged, not assumed).
- Winner score (D5): follower-aware blend. View component uses 2x trailing-10-post median per
  platform; warm-up: 3<=N<10 uses 2x median(N); N<3 flags none. Backfill (D8) skips most warm-up.
- Logging window: views logged at ~48h, refreshed ~7d (consistent so comparisons are fair).
- A's done test: idea -> make-shootable -> queued -> posted (1 post row/platform) -> logged ->
  winner flagged -> one-click 3 constrained clones, multi-device via Supabase.

## NOT in scope (considered, deferred)
- Full media/asset storage (raw clip uploads, B-roll library) — heavy, off-bottleneck (D9). Link field only.
- Scheduled/push Daily Briefing (email/cron) — needs always-on host + headless auth (D6). Button instead.
- Instagram Graph API auto-ingest — conditional on B0; manual log is enough for 10k.
- TikTok API — manual log indefinitely (strict approval, analytics live in TikTok's tab).
- Ranking by format/hook-pattern rather than lane — refinement once data shows lane is too coarse (TODO).

  one-time import of existing `cos_ideas` into the `ideas` table. Move dedupe server-side.
- `src/lib/claude.ts` `generateText()` — subscription Claude. Powers make-shootable + batch button. Reuse as-is.
- `src/components/IdeaGenerator.tsx` — reused for generation + make-shootable + clone-loop seeding.
- `src/components/HookGenerator.tsx` — reused for hook/caption on shootable conversion.
- `src/app/api/trends/route.ts` — Reddit/Google trends; the cold-start ranking fallback. Reuse.

## Failure modes (new codepaths)
- Supabase unreachable (D2): make-shootable / batch button must surface a clear error, not hang
  or silently lose data. NEEDS: error path + test. Currently no test, no handler -> CRITICAL GAP until built.
- Claude subprocess error (existing handler in generateText): surfaced via route 500 + UI copy. Covered.
- Winner score with sparse data: warm-up rules (D5) prevent false winners. NEEDS: unit tests at N=2,3,10 boundaries.
- Follower-delta attribution ambiguity (D5): two posts same day -> delta is shared/ambiguous.
  Handle as "candidate" not certain winner; NEEDS: test + explicit UI label.

Folded (non-contested correctness, no separate decision):
- **Navigation:** top-level Today / Bank / Results (replaces the old Ideas/Captions 2-tab). Bottom
  nav on mobile, header tabs on desktop (existing pattern).
- **Interaction states table** (what the user SEES):

  | Surface | Loading | Empty | Error | Success |
  |---|---|---|---|---|
  | Pace line | skeleton bars | "No history — backfill to light up winners" + Backfill CTA | "Couldn't reach DB; data safe; Retry" | ahead/behind delta + sparkline |
  | Today-to-film | spinner "Writing hooks… (a few sec)" | "Nothing queued — Generate tomorrow's batch" + CTA | inline error + Retry | 2-3 hook cards |
  | Idea bank | row skeletons | "Bank's empty — generate or backfill" | inline error | list + status pills + winner star |
  | Results log | — | "No posts logged yet" | save error + Retry | row appended, winner star animates in |
  | Batch button | button -> spinner, disabled | (n/a) | toast "generation failed; Retry" | queue fills |

- **Pace line framing:** show "X to go today" (per the ~131/day pace), not just a red "behind" —
  drives action instead of demoralizing. Red only when meaningfully off-pace.
- **Winner badge:** amber star on bank rows + a subtle one-time animation when a post crosses the
  winner threshold (the reward moment).
- **AI-slop fixes:** bottom-nav uses real line icons, NOT emoji; keep Geist (no system-font stack);
  no 3-col grid / centered-everything / purple.
- **Accessibility:** 44px min touch targets (status pills/chips currently too small), body text
  contrast >=4.5:1 (audit the zinc-500 muted text), visible focus states, ARIA label on the pace
  line + sparkline (it conveys data), keyboard nav for the queue/bank lists.

These add to the build: T6/T9 absorb the states + nav-icons + a11y + pace framing; a new
**T11 (P2)** covers the desktop two-pane layout.
